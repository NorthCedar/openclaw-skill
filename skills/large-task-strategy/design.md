# Large Task Strategy — Top-Level Design

## 定位

多 agent 编排框架，用于将超出单 agent 能力范围的任务拆分为可管理的子任务并行/串行执行。

## 解决的问题

| # | 问题 | 表现 |
|---|------|------|
| 1 | 上下文过载 | 单 agent 无法处理超大代码库/文档/数据源（>500行即显著降质） |
| 2 | 主进程阻塞 | 耗时操作占用主进程，导致无法响应用户 |
| 3 | 单一视角局限 | 复杂问题需要多角色（安全/性能/可读性）从不同维度审视 |

## 架构模型：扁平多叉树

```
         主进程（根节点 = 唯一调度者）
        /    |    |    \
      A      B    C    D（第1轮子进程）
     ✔️     ✔️   ✔️   ⚠️ NEEDS_SPLIT
                       |
                  主进程读取拆分建议，重新派发
                   /    |    \
                 D-1   D-2   D-3（第2轮子进程）
                 ✔️    ✔️    ✔️
```

### 节点角色

| 角色 | 身份 | 数量 | 核心特征 |
|------|------|------|---------|
| 主进程 | 根节点 | 1 | 唯一调度者，拥有所有派发权和写 shared.md 的权限 |
| 子进程（叶子） | 叶子节点 | N | 直接完成任务，输出产出文件 |
| 子进程（子树根） | 非叶子节点 | 0~M | 评估后发现任务需拆分，输出拆分建议 |

### 核心约束

- **调度权唯一**：只有主进程可以 spawn 子进程
- **子进程有判断权，无派发权**：可以判断任务是否需要拆分，但不能自行创建子进程
- **树的生长由主进程驱动**：子树根节点只提供建议，实际拆分和派发由主进程执行

## 进程职能划分

### 主进程（根节点）

**独占能力（只有主进程可以做）：**

| 能力 | 说明 |
|------|------|
| spawn 子进程 | 唯一可以 `sessions_spawn` 的角色 |
| 写 shared.md | 注册条目、更新所有状态（全局唯一写者） |
| 写 .meta.json | 创建 + 更新任务元状态 |
| 写 merged.md | 合并最终产出 |
| 管理 cron | 创建/删除 keepalive |
| 清理任务目录 | `rm -rf memory/wip/{task-id}/` |

**调度能力：**

| 能力 | 说明 |
|------|------|
| 侦察 | Phase 1 建立代码地图 |
| 任务拆分 | 分析依赖，决定并行/串行，分配子任务 |
| 分轮调度 | 按依赖图批量派发 |
| 验证产出 | 每个子进程完成后检查输出文件 |
| 状态同步 | 读取子进程产出文件标记 → 更新 shared.md registry |
| 处理 NEEDS_SPLIT | 读取拆分建议 → 注册新条目 → 派发 |
| 合并结果 | 读取各子进程产出 → stitch → 输出 |
| 向用户报告 | 每轮完成后报进度 |

### 子进程 — 叶子节点（直接完成）

| 能力 | 说明 |
|------|------|
| 读 shared.md | 确认自己的任务定义（只读） |
| 读其他 agent 产出文件 | 获取依赖知识（只读） |
| 执行任务 | 分析代码/写文档/审查等实际工作 |
| 写自己的产出文件 | `agent-{name}.md`，末尾标记 `## DONE` |

| 禁止 | 原因 |
|------|------|
| ❌ spawn 子进程 | 无派发权 |
| ❌ 写 shared.md | 只有主进程可写 |
| ❌ 写其他 agent 的文件 | 所有权隔离 |

### 子进程 — 子树根节点（NEEDS_SPLIT）

| 能力 | 说明 |
|------|------|
| 读 shared.md | 确认自己的任务定义（只读） |
| 读其他 agent 产出文件 | 获取依赖知识（只读） |
| 评估任务复杂度 | 判断是否需要拆分 |
| 写自己的产出文件 | `agent-{name}.md`，包含拆分建议，末尾标记 `## NEEDS_SPLIT` |

| 禁止 | 原因 |
|------|------|
| ❌ spawn 子进程 | 无派发权 |
| ❌ 写 shared.md | 只有主进程可写 |
| ❌ 写其他 agent 的文件 | 所有权隔离 |
| ❌ 自行执行拆分后的子任务 | 必须回报主进程 |

## 协议清单

### 协议 1：Task Registry 协议

**目的**：主进程与子进程之间的任务分配契约

**规则**：
- shared.md 是全局唯一协调文件，**只有主进程可写**
- 子进程只读 shared.md 获取自己的任务定义
- 主进程在派发前必须先注册条目
- Registry 包含：Agent 名、子任务、输入、输出路径、依赖、状态

**shared.md 结构**：
```markdown
# Task: {task-id}

## Context
用户任务描述（原始需求 1-3 句）
关键约束/规范/输出格式要求

## Task Registry
| Agent | Sub-Task | Input | Output | Depends | Status |
|-------|----------|-------|--------|---------|--------|
| agent-a | ... | ... | agent-a.md | - | ✅ done |
| agent-b | ... | ... | agent-b.md | agent-a | 🟡 running |
```

### 协议 2：产出文件协议

**目的**：子进程向主进程传达完成状态和结果

**规则**：
- 每个子进程有且仅有一个输出文件：`memory/wip/{task-id}/agent-{name}.md`
- 只有本子进程可写该文件，其他进程只读
- 文件末尾必须有状态标记（主进程据此判断状态）：
  - `## DONE` — 任务完成
  - `## NEEDS_SPLIT` — 需要拆分，文件包含拆分建议
  - `## FAILED` — 执行失败，文件包含失败原因

**主进程状态同步流程**：
```
子进程写完产出文件（含末尾标记）
  → 主进程检测到子进程结束
  → 主进程读取产出文件，检查末尾标记
  → 主进程更新 shared.md 中对应条目的 Status 列
```

### 协议 3：NEEDS_SPLIT 重调度协议

**目的**：子树根节点回报拆分建议后，主进程如何处理

**流程**：
1. 主进程检测到 agent-X 产出文件含 `## NEEDS_SPLIT`
2. 读取 `agent-X.md` 中拆分建议（建议格式见下方）
3. 更新 agent-X 在 registry 中的状态为 `⚠️ needs_split`
4. 根据建议注册新条目：`agent-X-1`、`agent-X-2`、`agent-X-3`...
   - 保留父级前缀便于追溯来源
   - Depends 列：如果新子任务需要 agent-X 的侦察信息 → 填 `agent-X`
5. 派发新子进程执行
6. agent-X.md 作为侦察记录保留，清理阶段随目录一起删除

**拆分建议格式**（子进程在产出文件中输出）：
```markdown
## NEEDS_SPLIT

### 拆分原因
任务涉及 3 个独立模块，总计 1200 行，超过单 agent 处理能力

### 建议拆分
| Sub-Task | Input | Depends |
|----------|-------|---------|
| 分析 auth 模块 | /path/auth.go:1-400 | - |
| 分析 handler 模块 | /path/handler.go:1-350 | - |
| 分析 middleware | /path/middleware.go:1-450 | - |
```

### 协议 4：依赖读取协议

**目的**：子进程如何获取其他子进程的知识

**规则**：
- shared.md 不承载背景知识，只有任务上下文和 registry
- 子进程需要其他子进程的知识时，直接读取对方的产出文件
- Task Registry 的 Depends 列标注依赖关系
- 主进程根据 Depends 列排轮次：有依赖的排在后面

**约束**：
- 只读，不可修改其他子进程的产出文件
- 后轮子进程的 task 中需明确指定读取哪些产出文件

### 协议 5：验证协议

**目的**：主进程确认子进程真正完成了任务

**三层防御**：
1. **Timeout** — 每个 spawn 必须有 `runTimeoutSeconds`
2. **文件检查** — `wc -l` + `head -5` + `grep "## DONE"`
3. **假完成检测** — 返回文本含"let me"/"接下来我"等 → 必须验证文件

**失败处理**：
- 同一任务失败 2 次 → 主进程自行接管执行
- 失败后主进程更新 registry 状态为 `❌ failed`

### 协议 6：多角色审查协议

**目的**：多个子进程从不同视角分析同一内容

**流程**：
1. 主进程为同一输入文件注册多个不同角色的子进程
2. 每个子进程的 task 中指定其审查角色和关注点
3. 各子进程独立输出分析结果（互不依赖，同轮并行）
4. 主进程合并多角色结论：
   - 共识部分 → 直接采纳
   - 冲突部分 → 主进程综合判断或上报用户

**角色示例**：
```
agent-security: 关注认证/授权/注入/数据泄露
agent-perf: 关注 N+1 查询/内存分配/并发瓶颈
agent-readability: 关注命名/结构/注释/可维护性
```

### 协议 7：进度报告协议

**目的**：让用户了解任务执行状态

**规则**：
- 每轮子进程完成后，主进程向用户报告进度
- 报告格式：完成了什么 / 发现了什么 / 下一步做什么
- NEEDS_SPLIT 时告知用户：某子任务需要进一步拆分，正在重新调度
- 最终合并完成后交付结果

## 数据流总览

```
用户需求
  ↓
主进程侦察 → map.md
  ↓
主进程拆分 → 注册到 shared.md
  ↓
主进程派发子进程（第1轮并行）
  ↓
子进程执行 → 写 agent-{name}.md（末尾 DONE/NEEDS_SPLIT/FAILED）
  ↓
主进程验证产出文件 → 更新 shared.md 状态
  ↓
如有 NEEDS_SPLIT → 读建议 → 注册新条目 → 派发（第2轮）
如有 Depends → 等依赖完成 → 派发后轮
  ↓
所有子进程完成 → 主进程合并 → merged.md
  ↓
交付结果 → 清理任务目录
```

## 文件权限矩阵

| 文件 | 主进程 | 子进程（本 agent） | 子进程（其他 agent） |
|------|--------|-------------------|---------------------|
| shared.md | 读写 | 只读 | 只读 |
| .meta.json | 读写 | — | — |
| map.md | 读写 | 只读 | 只读 |
| agent-{name}.md | 只读 | 读写 | 只读 |
| merged.md | 读写 | — | — |
| .shared.lock | — | — | — |

> 注意：由于只有主进程写 shared.md，不再需要 flock 并发锁。`.shared.lock` 文件从设计中移除。
